# Self-Evolution Skill

## 触发条件

当以下任一条件满足时，进入自进化分析：
- 同一任务连续失败 ≥ N 次
- 重复出现相同的 error pattern

## 分析流程

1. **收集上下文**：读取结构化日志，理解失败的具体场景
2. **定位问题**：只允许输出一个"改动假设"（single hypothesis），定位到单一问题点
3. **选择改动方式**（每次仅允许三选一）

## 三种改动方式

### 方式 1：修改 bia.md（行为纠偏规则）

适用场景：发现自己在某类任务上反复犯错，需要添加持久的行为修正。

操作：使用 `bia_update` 工具，追加一条规则到 bia.md。

规则格式：
```
- [场景] 当...时，必须...
```

限制：每次只允许添加或修改**一条**规则。

**生效方式**：即时生效 — 每次 chat 都会从磁盘重新读取 bia.md，无需重启。

### 方式 2：修改 skill（局部优化）

适用场景：某个 skill 的指令不够明确或有缺陷，导致执行偏差。

操作：使用 `edit_file` 修改对应 SKILL.md 的内容。

限制：每次只允许修改**一个** skill 的**一处**内容。

**生效方式**：即时生效 — `skill_view()` 调用时从磁盘加载，无需重启。

### 方式 3：增加 workflow（最小新增）

适用场景：现有工具无法完成某类多步操作，需要编排已有工具组成新的 workflow。

操作：使用 `write_file` 在 workflows 目录下创建新的 workflow 文件。

限制：
- 每次只允许新增**一个** workflow，且必须是最小实现
- workflow 只能编排已有工具，不能新增 tool（Function Calling 工具）

**生效方式**：需要重启 agent — Python 模块在启动时通过 importlib 加载并缓存在内存中，新增的 workflow 只有重启后才会被发现和注册。

**利弊说明**：
- 即时生效的方式 1 和 2 — 适合快速迭代验证，改完即可在同一会话内跑 benchmark
- 需重启的方式 3 — 流程更长，但 workflow 涉及更复杂的编排逻辑，重启后稳定性更好
- 如果用户在对话中进行了方式 3 的改动，**必须提醒用户重启 agent** 后才能生效

## 评估机制

改动后，使用固定的 benchmark 任务（3~5 个）验证：

1. 运行 benchmark 任务
2. 对比改动前后的执行结果（success/fail 或 score）
3. 决策规则：
   - 提升 → 保留改动
   - 无提升 / 不稳定 → 回滚改动（使用 edit_file 或 bia_update 恢复）
   - **禁止多改动叠加**，必须基线回归后再试下一种
